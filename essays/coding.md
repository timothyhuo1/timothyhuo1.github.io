A tool that I never really thought coders should have are coding standards. Coding standards are a way to organize and help a person code in a consistent matter. As a result, catching errors are less troublesome and will even prevent errors from happening. There are many rules to follow, such as spacing between mathematical equations, but in the long run, it has a huge payoff. Personally, when I code long projects that take weeks to finish, I tend to forget what parts of my code do and how it works. By implementing coding standards, it makes life easier for me to trace, ultimately saving time and effort. This also implies to group projects and displaying your work. Having a consistent coding standard in a group project benefits everyone and allows for quality code. Working together on a particular part or asking for help becomes more efficient when having a coding standard. This is because it makes code more readable and understandable. Just like if you have sloppy handwriting or code in this case, it makes it harder for others to help or understand what you are trying to do. 

## Benefits of Coding Standards
I believe that coding standards are not only a tool but a method of learning a programming language. As said before, coding standards help catch and prevent errors. In many cases, the mistakes that I make are those that I repeat throughout my codes. Some programs help monitor your code and indicate if something is wrong. Usually, this indicator stands out, so coders tend to fix it and not repeat it. Also, it helps beginners get used to strict notation. For example, case sensitivity and spelling when naming and calling variables and knowing every open bracket or parentheses will have a closing one. The indicator also provides solutions to help those who are learning the language to explore and understand how it works. Even without programs, just following the coding standards will help strengthen the foundation in the language.  

## My Experience So Far With ESLint
During this week, I began using ESLint with IntelliJ, and it took me a while to set it up and operate. At first, learning all the coding standards is frustrating, and the indicators are a bit annoying. However, after the first couple of projects, I got used the coding standards and started to expect what I needed to do right away, so the errors don’t show up. For ESLint, when you fix all the warnings and errors, you display a green checkmark. I ran into some minor problems with ESLint, such as with underscore. There are a couple of procedures to activate ESLint, and it took me a while until I knew what to do. Hopefully, over time and more practice, I’ll get the hang of it. So far, I believe that even with prior coding experience, I can develop some good practices that can improve my style of coding.
